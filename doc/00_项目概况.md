# 羽圈 App - 项目概况

## 项目基本信息

| 项 | 内容 |
|-----|------|
| **项目名称** | 羽圈（YuCircle） |
| **项目描述** | 垂直于羽毛球爱好者的社交平台 + 资讯应用 |
| **目标用户** | 打羽毛球的爱好者、专业运动员 |
| **主要功能** | 社区论坛 + 咨询资讯（初期以社区为主） |

---

## 核心价值定位

### 解决的痛点
- 打羽毛球的社群粘性高，但缺乏交流平台
- 现有平台（虎扑等）内容广泛，缺乏垂直深度
- 打球不爽时无处申诉/举报
- 需要一个专属羽毛球爱好者的圈子

### 解决方案
- 提供专业的社区论坛（类似虎扑社区）
- 聚焦羽毛球内容，深度运营
- 支持用户举报、审核机制
- 逐步补充赛事、运动员等资讯模块

## 项目范围定义

### MVP（最小可行产品）包含内容
✅ **社区核心**
- 用户认证系统
- 发帖/回复/评论
- 点赞/举报功能
- 个人中心

✅ **辅助功能**
- 话题分类
- 用户搜索
- 用户关注

❌ **暂不包含**
- 赛事数据模块（数据源待定，预留接口）
- 实时消息（后期扩展）
- 装备交易等其他功能

## 技术架构总览

```
┌─────────────────────────────────────────────┐
│         移动端（Flutter）                     │
│  iOS / Android 统一开发                       │
│  ├─ Material Design UI                        │
│  ├─ GetX 状态管理                             │
│  └─ Dio 网络请求                              │
└────────────────┬────────────────────────────┘
                 │ HTTP/REST API
┌────────────────▼────────────────────────────┐
│      后端服务（Spring Boot + MySQL）          │
│  ├─ JWT 身份认证                             │
│  ├─ MyBatis-Plus ORM                        │
│  ├─ 文件上传存储                              │
│  └─ 业务逻辑层                               │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│      数据层（MySQL）                         │
│  ├─ 用户表                                    │
│  ├─ 帖子/评论表                               │
│  ├─ 点赞/举报表                               │
│  └─ 赛事表（预留）                            │
└─────────────────────────────────────────────┘
```

### 技术栈选型

**前端（Flutter）**
- **状态管理**: GetX（快速开发、学习成本低）
- **网络请求**: Dio（强大、易用）
- **图片处理**: image_picker + cached_network_image
- **UI框架**: Material Design + 自定义组件

**后端（Java）**
- **框架**: Spring Boot 3.x
- **ORM**: MyBatis-Plus
- **认证**: JWT
- **文件存储**: 本地文件系统（初期）

**数据库**
- **数据库**: MySQL 8.0+
- **核心表**: 用户、帖子、评论、点赞、举报、赛事

## 关键决策记录

1. **优先社区、暂延赛事** - 社区是核心，赛事数据源不确定，后期补充
2. **不做实时功能** - 简化复杂度，初期采用轮询或定时刷新
3. **单文件存储** - 初期本地文件系统，后续可迁移到 OSS
4. **手机号唯一性** - 登录唯一标识，确保用户管理清晰
5. **先做后端、再做前端** - 确保 API 接口设计稳定
